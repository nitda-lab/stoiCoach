# Stoic — 理想の自分になるための生活サポートAI 設計書

- 日付: 2026-06-25
- ステータス: 設計合意済み（MVP）
- 対象: MVP（コア体験の検証）

## 1. 概要・ビジョン

ユーザーが「理想の自分」になるための生活（習慣・ルーティン・タスク・やめること・中長期の目標）を、
**AIとの対話を通じて構造化・管理・追跡**できるWebサービス。

全体ビジョンには次の独立サブシステムが含まれる:

1. AIチャット基盤（自然言語で意図を伝える）
2. 動的な管理項目の生成（AIが対話から管理項目を作成・操作）
3. トラッキング & ストイックポイント（日/週/月/累計）
4. ランキング & 他ユーザーとの競争（マルチユーザー、ソーシャル）

本設計書のスコープは **MVP = サブシステム1〜3の最小実装** に絞る。
ランキング(4)は将来拡張とし、データ設計だけ将来集計可能な形にしておく。

## 2. MVPのコア体験

**「チャットで目標を伝えると、AIがパーソナライズした管理項目を動的に生成し、
ダッシュボードで追跡できる」** ことをコアに据える。

- 項目タイプは4つに固定せず、AIが対話の中で柔軟に「追跡方法（プリミティブ）」を選んで組み立てる。
- ストイックポイントの重みもAIが難易度に応じて提案し、ユーザーは必要なら調整できる。

## 3. 中核メカニズム: ツール実行型エージェント（案A採用）

AI（nanoGPT経由）が会話しつつ、必要に応じて**ツール（function calling）**を呼び出し、
バックエンドがその呼び出しをDBに反映する。作成・修正・削除・チェックすべてを同じ会話インターフェースで扱える。

公開するツール（API側で実装）:

- `create_item` — 管理項目を作成（`tracking_type`, `config`, `point_weight` を含む）
- `update_item` — 既存項目を修正（周期変更、重み調整、リネーム等）
- `complete_item` — 達成を記録（Completionを1件作成 = ポイント発生）
- `archive_item` — 項目をアーカイブ/終了
- `list_items` — 現在の項目を取得（AIが状況を把握するため）

AIがツールを呼ぶと、チャット内に確認チップ（例 `✓ 朝ランを追加しました`）＋カードを表示する。

### 不採用案
- 案B（チャット→構造化JSON一括生成）: 初期生成は楽だが、既存項目の継続的な部分修正に弱く、コア価値を満たせない。
- 案C（固定フォーム + AI補助）: 堅実だが「AIが動的生成」というコンセプトが薄れる。

## 4. 画面レイアウト

チャット画面とダッシュボード画面を**独立したルート**に分ける。
ナビゲーションは**全デバイスでトップナビ**（ヘッダー直下のタブ切替）。
※ 後から「モバイルのみボトムナビ」に変更するのは小さな変更で可能。

```
 ナビ: [ 💬 チャット ]   [ 📊 ダッシュボード ]   [ 👤 ]

┌── チャット画面 ──────────────┐     ┌── ダッシュボード画面 ───────┐
│ AI: おはよう。今日は何を…     │     │ 🔥 累計 1,240pt            │
│ You: 朝ランを習慣にしたい     │     │ [ 今日 ] 週  月            │ ←期間タブ
│ AI: 毎日でいきましょう。      │     ├───────────────────────────┤
│  ┌─ ✓ 追加しました ───────┐  │     │ ☐ 朝ラン      🔥7  +10pt │ ←繰り返し
│  │ ☐ 朝ラン 毎日 +10pt    │  │ ←  │ ☐ 甘いもの我慢 3日   +8  │ ←回避
│  └───────────────────────┘  │ 必要時│ ☐ 請求書を送る 期限明日   │ ←一回限り
│ You: 進捗どう？              │ だけ │ ▓▓▓░░ 簿記2級 60%  +20  │ ←進捗
│ AI: 今日はあと2つ。         │ インラ│                           │
│  ┌─ 残タスク ───────────┐  │ イン │  各カードを手動でチェック/  │
│  │ ☐ 甘いもの我慢        │  │ 表示 │  編集も可能               │
│  └───────────────────────┘  │     │                           │
│ [ メッセージ入力…    送信 ] │     │                           │
└─────────────────────────────┘     └───────────────────────────┘
```

- **チャット画面**: AIとの会話。AIがツール実行したら、その項目カードだけをインライン表示。一覧はダッシュボードへ。
- **ダッシュボード画面**: 「今日 / 週 / 月」タブ＋累計pt。追跡プリミティブごとにカードの見た目が変わる
  （チェックボックス / 日数カウンタ / 期限ラベル / 進捗バー）。手動チェック・編集も可能。
- 両画面は同じDBを参照するため、チャット操作はダッシュボードに即反映（双方向）。

## 5. データモデル

```
User ──┬─< Conversation ──< Message        (チャット履歴)
       │
       └─< Item ──< Completion             (管理項目と達成記録)
```

### Item（管理項目）— AIが生成する中核オブジェクト
1テーブルで4タイプを表現（`tracking_type` + `config` JSON）。

| フィールド | 内容 |
|---|---|
| id | 主キー |
| user_id | 所有ユーザー |
| title | 例「朝ラン」 |
| tracking_type | `recurring` \| `one_time` \| `avoidance` \| `progress` |
| config | JSON（tracking_typeごとに中身が変わる） |
| point_weight | AIが提案した重み（例 10） |
| status | `active` \| `archived` \| `done` |
| created_at | 作成日時 |

`config` の中身（tracking_typeで分岐）:

- `recurring` : `{ cadence: "daily"|"weekly"|..., target_per_period: 1 }`
- `one_time`  : `{ due_date: "2026-06-26" }`
- `avoidance` : `{ since: "2026-06-25" }`（我慢開始日。日数計算用）
- `progress`  : `{ milestones: [...], percent: 60 }`

### Completion（達成記録）— チェック1回 = ポイント発生源

| フィールド | 内容 |
|---|---|
| id | 主キー |
| item_id | 対象Item |
| user_id | 所有ユーザー |
| date | その達成が属する日 |
| points_earned | その時の付与pt（重みのスナップショット） |
| created_at | 作成日時 |

### 設計の要点
- ポイントは集計値を持たず、**Completionを「発生イベント」として記録**。
  日/週/月/累計は `Completion` を期間で `SUM(points_earned)` するだけ。
  将来のランキングも同じ集計で算出可能。
- `points_earned` は達成時の重みを**スナップショット保存**（後で重みを変えても過去ptは不変）。
- ストリーク・進捗%は `Completion` 履歴から計算、または `config` に保持。

## 6. システム構成

**スタック: React/Vite SPA + Hono(API) + Neon(Postgres) + Clerk(認証) + nanoGPT(LLM)**

```
[React/Vite SPA] ──Clerkで認証──┐
   │ ① 発話送信(JWT付き)          │
   ▼                            │
[Hono API サーバー]  ← nanoGPTキーをここに秘匿、ClerkでJWT検証
   │ ② 履歴+ツール定義 → nanoGPT
   ▼
[nanoGPT] (OpenAI互換)  ③ 返答 or ツール呼び出し
   ▼
[Hono] ④ ツールを実行 → [Neon Postgres] にcreate/update/complete
   │ ⑤ 必要なら結果を戻して最終文生成
   ▼
[SPA] チャット返答+カード / ダッシュボード再取得
```

### 技術選定
- **フロント**: React + Vite + TypeScript。サーバ状態取得は TanStack Query。
- **APIサーバー**: Hono。nanoGPTのAPIキーを秘匿し、ClerkのJWTを検証。ツール（function calling）を実装。
- **DB**: Neon（サーバーレスPostgres）。ブランチ機能を開発/テスト用DBに活用。
- **認証**: Clerk。
- **LLM**: nanoGPT（OpenAI互換API）。capableなモデル（Claude/GPT系）にルーティングし、
  function calling で上記ツールを公開。構造化出力で管理項目を生成。

### ポイント設計
- `create_item` の引数に `point_weight` を含め、AIが難易度から提案。
- ユーザーが「重すぎ/軽すぎ」と言えば `update_item` で調整。

## 7. エラーハンドリング
- **nanoGPT呼び出し失敗/タイムアウト**: チャットに再送を促すメッセージ。ツール実行前の失敗ならDBは無変更（副作用なし）。
- **ツール引数のバリデーション**: `tracking_type`が未知、`point_weight`が負などはAPI側で検証。
  不正なら実行せずAIに差し戻すかエラー表示。
- **楽観的更新の整合**: ダッシュボードの手動チェックは optimistic update。失敗時はロールバック。
- **認証切れ**: Clerkトークン失効時はログイン画面へ誘導。

## 8. テスト方針
- **ユニット**: ポイント集計（日/週/月/累計のSUM）、ストリーク計算、`config`バリデーション。
- **APIレベル**: 各ツール（create/update/complete/archive）がDBを正しく変更するか（テスト用Neonブランチを使用）。
- **AI連携**: nanoGPT応答をモックし、「ツール呼び出しJSON→DB反映」経路をテスト（実API呼び出しは最小限）。
- **E2E（軽め）**: 「チャットで項目追加→ダッシュボードに出る→チェックでpt加算」の主要動線。
